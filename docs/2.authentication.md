# Authentication

## Current state: fake, client-only

`pages/login.vue` does **not** call an API. On submit it builds a fake `user` object from the
email address, sets an `expiry` 1 day out, and writes it straight into `useAuth()`. This is a
placeholder to make the rest of the dashboard (which assumes a logged-in user) usable out of
the box — it is the first thing to replace with a real backend call.

```ts
const formHandler = defineFormHandler({
  state: formModel,
  schema: formSchema,
  async onSubmit(data) {
    const user = { id: ..., name: data.email.split('@').at(0)!, email: data.email };
    const isValidAuthState = await auth.setState({
      user,
      token: ...,
      permissions: ['dashboard'],
      expiry: ... // now + 1 day
    });
    // ...
    window.location.replace('/');
  }
});
```

## `useAuth()`

`app/composables/auth.ts` wraps a VueUse `useStorage` value (key `_auth`) holding
`{ user, token, permissions, expiry }`, validated against a Zod schema in `app/utils/auth.ts`.
Key methods:

- `auth.setState(payload)` — validates `payload` against the schema and persists it if valid;
  returns whether it was accepted.
- `auth.valid()` — true only if state exists, parses against the schema, **and** `expiry` is
  still in the future. This is the single source of truth for "is the user logged in."

There is no token refresh, no server-side session check, and no revocation — `valid()` is a
local, offline check. Anything security-sensitive must be enforced by a real backend once one
exists; the client-side check is a UX gate, not a security boundary.

## Route protection

Two middlewares, applied per-page via `definePageMeta({ middleware: '...' })`:

| Middleware | Behavior |
|---|---|
| `auth` | If `useAuth().valid()` is false, redirect to `/login`. Use on protected pages. |
| `unauth` | If `useAuth().valid()` is true, redirect to `/`. Use on `/login` so an already-logged-in user can't see it again. |

New protected pages opt in like this — no manual check needed inside the page:

```ts
definePageMeta({ middleware: 'auth' });
```

## Logout

`layouts/default.vue` calls `useRequest('/logout')` on logout — note this **does** go through
the real request helper (unlike login), so once a backend exists, a real `/logout` endpoint is
expected to exist and be called. Today it will hit the dev fake-API middleware or 404 outside
dev, but the auth state is cleared client-side regardless via `location.replace('/login')`
after the request settles.

## What to change for a real backend

1. Replace the fake `onSubmit` in `login.vue` with a real `useRequest('/login', { method: 'POST', body: ... })` call, and populate `user`/`token`/`permissions`/`expiry` from the response instead of fabricating them.
2. Add token refresh if your backend issues short-lived tokens — `useAuth()` has no hook for this today.
3. Keep `valid()`'s shape (`{ user, token, permissions, expiry }`) or update the Zod schema in `app/utils/auth.ts` to match your real payload — everything downstream (middleware, `AppSidebar`, etc.) reads through `useAuth()`, not the raw storage key.
