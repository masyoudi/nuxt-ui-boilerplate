# Forms & Validation

Every form in this boilerplate follows the same pattern: a Zod schema + `defineFormHandler()`
(`app/utils/form.ts`), wired into Nuxt UI's `<UForm>`. Use this pattern for new forms rather
than hand-rolling submit/validate/error logic per page.

## The pattern

```ts
import z from 'zod';

const formSchema = z.object({
  email: z.email('Please enter email address'),
  password: z.string().refine((val) => val.trim().length > 0, 'Please enter password')
});

const formModel = reactive<z.output<typeof formSchema>>({
  email: '',
  password: ''
});

const formHandler = defineFormHandler({
  state: formModel,
  schema: formSchema,
  async onSubmit(data) {
    // data is typed as z.output<typeof formSchema>
  }
});
```

```html
<UForm v-bind="formHandler" class="space-y-6">
  <UFormField label="Email" name="email">
    <UInput v-model="formModel.email" />
  </UFormField>
  <!-- ... -->
  <UButton type="submit" :loading="loading">Submit</UButton>
</UForm>
```

## What `defineFormHandler` gives you

- **Validation** — runs `schema.safeParseAsync` and maps Zod issues to the `{ name, message }`
  shape `<UForm>` expects, so field-level errors show up under the right input automatically.
- **Submit guarding** — `onSubmit` is a no-op while a previous submit is still in flight
  (checked via an internal `loading` ref), so double-submits from fast double-clicks are
  ignored for free.
- **Error UX** — `onError` scrolls to and focuses the first invalid field automatically
  (`scrollIntoView` + `focus`, ~150ms after scroll settles), so long forms don't leave the
  user hunting for what's wrong.
- **Dynamic schemas** — `schema` can be a function of the current state
  (`(data: D) => T`) instead of a static schema, for cases like "field B is required only if
  field A is set."

## Conventions to follow

- Field-level errors come from the Zod schema's messages — write user-facing messages in the
  schema itself (as in the example above), not as separate copy elsewhere.
- `name` on `<UFormField>` must match the schema key exactly — that's how errors get routed
  to the right field.
- `defineFormHandler` guards against double-submit internally, but it does **not** expose a
  `loading` state for your submit button — `login.vue` keeps its own local `loading` ref and
  sets it inside `onSubmit`. Follow that pattern: your own `loading` ref, toggled at the start
  and end of `onSubmit`, passed to the button's `:loading` prop.
- The returned `disabled` value is a snapshot (`toValue(options.disabled)`) taken once, not a
  reactive binding — if you need the form to disable/enable reactively, don't rely on this
  field alone; drive `<UForm>`'s `disabled` prop from your own reactive state instead.
