# ARCHITECTURE.md

High-level notes on how this project is put together and why. Read `AGENTS.md` first for
day-to-day conventions — this file explains the *shape* of the app.

## What this is

A **client-only (SPA) dashboard boilerplate**, not a full-stack app with a real backend.
`server/` is empty. The intent is: clone this, wire up your real API, keep the UI layer.

## Why SPA, not SSR

`nuxt.config.ts` sets `ssr: false`. This is deliberate — dashboards behind a login screen
gain little from server rendering (nothing to index, first paint behind auth anyway) and
lose the complexity of hydration/SSR-safe code. If you need SEO or public pages, the
maintainer keeps that as a **separate branch (`ssr`)** rather than a config flag in this one,
because SSR changes real patterns (no `window`/`document` in setup, `useAsyncData` semantics,
etc.), not just a toggle.

## Directory layout as an architecture, not just a listing

```
app/            ← Nuxt srcDir. Everything client-facing lives here.
shared/utils/   ← Code safe to run on both client and (future) server — pure functions only.
server/         ← Nitro backend. Empty today; this is where real API routes will live.
config/         ← Build-time config: env loading, color generation (not runtime app config).
public/         ← Static passthrough assets.
```

The split between `app/utils/` and `shared/utils/` matters: `shared/utils/` is for code with
zero DOM/browser dependency, because it's meant to be reusable once `server/` gains real
Nitro API routes. `app/utils/` can assume a browser.

## Request flow (today: fake data, tomorrow: real API)

There is no real backend yet. In development, `nuxt.config.ts` registers a **Vite middleware**
(`api-dev`) that intercepts `/api-dev` and returns data generated by `faker-generator.ts` via
`@faker-js/faker`. This exists purely so table/list/chart UI has something to render during
development.

```
Component → useRequest(path) → $fetch (ofetch) → [dev: Vite middleware /api-dev, fake data]
                                               → [prod: real API — not yet implemented]
```

`useRequest()` / `useRequestError()` (`app/utils/request.ts`) are the single chokepoint for
all HTTP calls — fixed timeout, no retry, and errors are funneled into a toast automatically.
When wiring a real backend, this is the one place that needs a `baseURL`, not every call site.

## Auth model

Auth is **client-side only** — there is no server session. `useAuth()`
(`app/composables/auth.ts`) persists `{ user, token, permissions, expiry }` (validated with a
Zod schema, `app/utils/auth.ts`) into browser storage via VueUse's `useStorage`. Validity is
just "does it parse and is `expiry` in the future" — there's no token refresh or server check.

Two route middlewares gate navigation instead of per-page checks:
- `auth` — redirects to `/login` if `useAuth().valid()` is false
- `unauth` — redirects to `/` if already authed (keeps a logged-in user off `/login`)

This means **route protection is a middleware concern, not a page concern** — a new protected
page opts in by declaring `middleware: 'auth'`, it doesn't reimplement the check.

## Rendering shell

`app.vue` → `<UApp>` (Nuxt UI's app-level provider for toasts/tooltips) → `<NuxtLayout>` →
`<NuxtPage>`. Two layouts:
- `default.vue` — the real dashboard shell: sidebar + header + logout modal. Reads/writes the
  page `<title>` (via `useHead` + a `MutationObserver`) to drive the header title, rather than
  each page passing a title prop down.
- `blank.vue` — no chrome, used for `/login`.

Sidebar collapsed/expanded state persists via a cookie (`sidebarmini`), not component state,
so it survives reloads and (if SSR is ever re-enabled) is available on first render.

## Theming pipeline

Brand colors are defined **once**, in `config/colors.ts`, as plain hex values. From there:
1. At build time, a Nuxt hook (`hooks: { 'build:before': createAssetColors }`) generates
   `app/assets/css/colors.css` — a full OKLCH shade ramp per color, written as CSS custom
   properties. This generated file is marked "don't modify" — treat it as build output, not
   source.
2. At runtime, `app.vue` reads a `__themecolor` cookie (user-picked accent color) and
   overrides the `--color-primary*` variables live via `chroma-js` + `theme-colors`, so users
   can pick a custom accent without a rebuild.
3. Component-level visual tweaks (not raw colors) live in `app.config.ts` (Nuxt UI slot/variant
   overrides) or `app/theme/*.ts` for larger per-component overrides.

So: **new brand color → edit `config/colors.ts`. New accent picker option → runtime cookie
logic in `app.vue`. New component look → `app.config.ts` or `app/theme/`.** These are three
different layers on purpose; don't collapse them.

## Not yet decided / left open

- No state management library beyond Vue's built-in `useState`/composables — no Pinia/Vuex.
  `useAuth()` shows the pattern the boilerplate expects for shared state.
- No test setup. Adding one is an open decision, not an oversight to "fix" silently.
- `server/` has no routes, no folder conventions established — first real API route sets the
  precedent, so keep it idiomatic Nitro (`server/api/<name>.ts`) rather than inventing structure.
